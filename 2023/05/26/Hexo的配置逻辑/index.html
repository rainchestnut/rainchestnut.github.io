Hexo + Github Pages 配置

本文档主要介绍如何部署 Hexo 的配置逻辑，网上分享怎么配置Hexo个人博客的教程很多，但很少有人记录为什么要这样配，这么做的逻辑是什么：

1.为什么固定为用户名.github.io 这种格式？

GitHub Pages 要求你的仓库名称为 username.github.io 这种格式，因为这是 GitHub Pages 的工作方式。GitHub 使用这种格式来确定哪个仓库应该被用来构建和发布你的 GitHub Pages 网站，简而言之，人家规定的。

2. Hexo如何将md文件放到github.io让我们能看到的？

Hexo 使用一个名为 hexo-deployer-git 的插件将你的网站部署到 GitHub。当你运行 hexo deploy 命令时，Hexo 会生成你的网站，并通过 Git 将生成的文件推送到你的 GitHub 仓库。GitHub Pages 会自动检测到这些新的文件，并将你的网站更新到最新的状态。

3. 为什么都配置了dns解析，将我们自己的域名映射到github.io上，还需要CNAME文件？

CNAME 文件在这个过程中起到了“桥梁”的作用。当你的域名被访问时，CNAME 文件在这个过程中的确起到了一个关键的角色。在 DNS 解析后，用户的请求会被导向到 username.github.io

  1. GitHub 会查看你的仓库中的 CNAME 文件来确定正确的自定义域名，并确保你的自定义域名正确显示在浏览器的地址栏中。CNAME 文件本身并不含有任何服务器的信息，它只是一个包含你的自定义域名的简单文本文件。它告诉 GitHub：当用户的请求到达 username.github.io 时，应该显示哪个自定义域名在浏览器的地址栏中。

  2. GitHub Pages 在服务端识别请求时，是根据仓库名称（如 username.github.io）来判断的，如果你直接输入这个地址访问，的确是可以找到对应的 GitHub Pages 仓库并正常访问。

但是当你使用自定义域名并且在 DNS 服务商那里将你的自定义域名解析到 username.github.io 时，GitHub Pages 在接收到请求后，会查看请求的 Host Header（即你的自定义域名），并尝试在所有的 GitHub Pages 仓库中寻找一个 CNAME 文件与之匹配的仓库，以决定显示哪个仓库的页面。

如果你没有在 GitHub 仓库中添加对应的 CNAME 文件，那么 GitHub Pages 无法找到一个与你的自定义域名匹配的仓库，因此会返回 404 错误。

这我看来这本质上是一个权限控制的问题，控制只有你所允许的自定义域名才能访问这里，正如github设置栏所言

Custom domains allow you to serve your site from a domain other than XXX.github.io.

这就是为什么当你使用自定义域名时，需要在 GitHub 仓库中添加一个 CNAME 文件。CNAME 文件起到了将你的自定义域名与你的 GitHub Pages 仓库关联起来的作用。
